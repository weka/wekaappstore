## Issue 1 — MCP inspect tools never call `load_incluster_config()`

**Severity:** Critical (blocks all cluster-inspection tool calls)
**Scope:** `mcp-server/tools/inspect_cluster.py`, `mcp-server/tools/inspect_weka.py`

### Evidence

Direct MCP invocation from the openclaw container:

```
curl -X POST http://localhost:8080/mcp \
  -H "Accept: application/json, text/event-stream" \
  -d '{"jsonrpc":"2.0","method":"tools/call","id":1,"params":{"name":"inspect_cluster","arguments":{}}}'
```

returned a response with these warnings:

```
HTTPConnectionPool(host='localhost', port=80): Max retries exceeded with url: /api/v1/namespaces
HTTPConnectionPool(host='localhost', port=80): Max retries exceeded with url: /apis/storage.k8s.io/v1/storageclasses
HTTPConnectionPool(host='localhost', port=80): Max retries exceeded with url: /api/v1/nodes
```

and every resource field (`cpu_nodes`, `gpu_total`, `visible_namespaces`, etc.) was `null` or empty.

### Root cause

`mcp-server/tools/inspect_cluster.py:104-105`:

```python
from webapp.inspection.cluster import collect_cluster_inspection
snapshot = collect_cluster_inspection()   # <-- missing load_kube_config
```

The underlying `collect_cluster_inspection()` in `app-store-gui/webapp/inspection/cluster.py:133-142` takes a `load_kube_config` callable argument and only invokes it when passed. Because the MCP wrapper doesn't pass one, the kubernetes-python client falls back to its internal default which is `localhost:80` (no config). Sibling tools (`status_tool.py:57`, `crd_schema.py:54`) correctly call `k8s_config.load_incluster_config()` before use. The inspect wrappers don't.

The same bug is presumed in `inspect_weka.py` (same calling pattern) — confirm during the v3.1 fix.

### Verification that the environment is not at fault

Running from inside the MCP sidecar:

```
python3 -c "from kubernetes import client, config; config.load_incluster_config(); print(len(client.CoreV1Api().list_node().items))"
```

returned `18` — proving the ServiceAccount, CA cert, token, and `KUBERNETES_SERVICE_HOST=10.100.0.1` env are all wired correctly. The bug is purely in the tool wrapper.

### Why Phase 13 validation missed it

`scripts/validate-phase13.sh --live` checked:
- `/health` returns 200
- RBAC: `kubectl auth can-i list nodes` (from the SA, not through the MCP server)
- `openclaw.json` content
- `SKILL.md` mount

It **never invoked an MCP tool over HTTP and verified the response body**. Phase 13 VERIFICATION.md correctly flagged these as `? HUMAN` gaps but the live human check didn't exercise the data path either.

### v3.1 fix

1. Edit `mcp-server/tools/inspect_cluster.py`: before calling `collect_cluster_inspection()`, call `k8s_config.load_incluster_config()` (with fallback to `load_kube_config()` for local dev, matching `status_tool.py` pattern).
2. Same fix for `mcp-server/tools/inspect_weka.py`.
3. Add a Phase 13/14-style test that calls each tool over HTTP and asserts non-null cluster data. This is the Nyquist gap — phase validation was signing off on tools that never actually ran.

## Issue 2 — Init container's openclaw.json is missing runtime config keys

**Severity:** High (agent cannot call any MCP tool until manual fix applied)
**Scope:** `k8s/agent-sandbox/openclaw-sandbox.yaml` init container `printf` command

### Evidence

The init container writes:

```json
{"mcp":{"servers":{"weka-app-store-mcp":{"url":"...","transport":"...","skill":"..."}}}}
```

Agent tool calls then fail with:

```
[tools] exec failed: exec host not allowed (requested node;
  configure tools.exec.host=sandbox to allow).
```

During this session, three additional config keys had to be set manually on the live pod to get the exec path working:

| Key | Value used | Purpose |
|-----|-----------|---------|
| `tools.exec.host` | `auto` | Allow MCP tool invocations to route to whatever host is available (without this, defaults reject `node` host) |
| `agents.defaults.sandbox.mode` | `off` (or unset) | Disable Docker-in-Docker sandboxing (container has no `docker` binary) |
| `models.providers.nim.*` | (full NIM provider block) | Model routing for NIM Llama 3.1 8B — not written by init container |
| `agents.defaults.model.primary` | `nim/meta/llama-3.1-8b-instruct` | Agent default model |

The live pod had these keys because **someone ran `openclaw configure` manually after Phase 13 deployment**. They are not captured in any version-controlled manifest. Every pod restart regenerates `openclaw.json` to just the mcp block, and the agent breaks again.

### Why Phase 13 validation missed it

The init container's `printf` was written when openclaw-gateway's defaults happened to work without explicit config. Newer OpenClaw versions (or this particular install) require explicit `tools.exec.host`. The gap is structural — nothing in the manifests documents that these extra keys are required.

### v3.1 fix

1. Extend the init container's `printf` (or move to a proper ConfigMap + init container copy pattern) to include `tools.exec.host`, `agents.defaults.sandbox.mode`, and model provider config.
2. Move secrets (NIM baseUrl if sensitive) to Secret refs using OpenClaw's provider builder pattern: `openclaw config set ... --provider-source env ...`.
3. Write a validation test that checks the generated `openclaw.json` contains all required keys before the gateway starts.
4. Evaluate switching from custom `printf` init container to a ConfigMap-based approach — version-controlled and easier to diff.

## Issue 3 — Small-model reliability for agentic tool use (Llama 3.1 8B)

**Severity:** Medium (user-experience, not a code bug)
**Scope:** deployment topology / model choice

### Evidence

Observed failure modes during the live chat attempt:

1. **Tool name hallucination.** The agent called `inspect_weka_storage` (doesn't exist) instead of `inspect_weka`. Logged at `2026-04-21T05:38:30.603Z [ws] → event agent ... meta=mcp inspect_weka_storage ...`.
2. **First-try wrong tool.** The agent called OpenClaw's built-in `memory_search` (which also fails — see Issue 4) before attempting any MCP tool, despite SKILL.md explicitly listing `inspect_cluster` as step 1.
3. **Error perseveration.** After a failed tool call, the agent stopped trying and regurgitated the error message pre-emptively on the next user turn — never attempting the tool again even after the underlying config was fixed. Evidenced by the OpenClaw logs showing no `[tools]` entry on the third retry, only streamed assistant text replaying the prior error verbatim.

These are consistent with known 8B-class model limitations for multi-step tool use.

### v3.1 options

1. **Upgrade to a larger NIM model** — Llama 3.1 70B or Llama 3.3 70B if the NIM endpoint serves one. Requires GPU capacity check.
2. **Use an external provider** (OpenAI, Anthropic) via OpenClaw's provider system. Requires API key in Secret + provider config.
3. **Tighter SKILL.md prompting** — more explicit tool-first instructions, example tool calls, smaller per-step scope. May help marginally with 8B but won't eliminate perseveration.
4. **Accept limitation and document it** — run E2E via `openclaw agent --local --message ...` CLI in a script rather than interactive chat.

Recommended: (1) or (2) — the whole point of E2E validation is proving the chat experience works.

## Issue 4 — OpenClaw built-in memory tool uses OpenAI embeddings with invalid API key

**Severity:** Low (degrades chat but doesn't block tools)
**Scope:** OpenClaw config defaults

### Evidence

`openclaw memory status --deep` returned:

```
Embeddings: unavailable
Embeddings error: openai embeddings failed: 401 {
    "error": {"message":"Incorrect API key provided: not-needed. ..."}
}
```

The agent calls `memory_search` on first user turn because the 8B model prefers memory lookups over external tools. The tool fails with a 401, the agent reports the failure to the user, and that "broken first tool call" kicks off the error-perseveration pattern in Issue 3.

Memory index is empty (0 files, 0 chunks) — even if embeddings worked, there's nothing to search.

### v3.1 options

1. **Disable the memory tool** via openclaw config (if there's a key — none found in 2026.3.23 CLI help).
2. **Configure a local embedding provider** — NIM may serve `text-embedding-*` models at a separate endpoint, or use sentence-transformers on CPU.
3. **Explicit prompting in SKILL.md** to tell the agent to skip memory_search for this workflow.

**Severity:** Low (working but 20+ versions behind)
**Scope:** image pin in `openclaw-sandbox.yaml`

### Evidence

