  <action>
    **SKILL.md** (at mcp-server/SKILL.md):
    Create the authoritative agent workflow document with these sections:

    1. **Overview** - Brief description of the WEKA App Store MCP server and what it enables
    2. **Workflow** - Numbered step-by-step:
       - Step 1: inspect_cluster (understand available resources)
       - Step 2: inspect_weka (check WEKA storage state)
       - Step 3: list_blueprints (find available blueprints matching cluster)
       - Step 4: get_blueprint (get full details of selected blueprint)
       - Step 5: get_crd_schema (get the WekaAppStore CRD spec for YAML generation)
       - Step 6: Generate YAML (agent generates WekaAppStore manifest using CRD schema)
       - Step 7: validate_yaml (validate the generated YAML)
       - Step 8: If validation fails, read errors, fix YAML, re-validate (max 3 attempts, then ask user for help)
       - Step 9: Re-run inspect_cluster (MANDATORY before apply - confirm resources unchanged)
       - Step 10: Show user the validated YAML, resource name, namespace, and deployment method. Wait for explicit approval.
       - Step 11: apply (with confirmed=true only after user approval)
       - Step 12: status (monitor deployment progress)
    3. **Validate-Retry Loop** - Explicit section: read validation errors, fix YAML, call validate_yaml again. Max 3 attempts. If still failing, present errors to user and ask for help.
    4. **Re-Inspect Before Apply** - Explicit rule: after validation passes and before calling apply, ALWAYS re-run inspect_cluster. Cluster state may have changed. If resources are now insufficient, inform user.
    5. **Negative Examples** - YAML snippets showing what NOT to do:
       - v1.0 field mistakes: blueprint_family, fit_findings (snake_case fields rejected by validator)
       - Wrong apiVersion (must be warp.io/v1alpha1 or similar)
       - Missing metadata.name
       - Skipping validation before apply
       - Applying without inspecting cluster first
       - Setting confirmed=true without user approval
    6. **Tool Reference** - Brief per-tool summary linking back to workflow steps

    **Tool Description Tuning** (all 8 tools in tools/*.py):
    Ensure every tool description has:
    - When to call it (relative to other tools)
    - What it returns
    - Explicit cross-references to other tool names
    - Safety warnings where applicable

    Specific additions per CONTEXT.md decisions:
    - apply_tool.py: Add "Before calling apply, re-run inspect_cluster to confirm cluster resources haven't changed since initial inspection."
    - get_crd_schema: Add sequencing hint that this should be called before generating YAML, after get_blueprint
    - validate_yaml: Add hint about the retry loop (fix and re-call if errors found)
    - status_tool: Add hint to call after apply to monitor progress
    - All tools: Ensure consistent "Sequencing:" line format

    Do NOT change any _impl() function logic or parameters - only modify the docstring/description strings inside @mcp.tool() decorators.
