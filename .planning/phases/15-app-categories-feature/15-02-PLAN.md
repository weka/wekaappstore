<objective>
Structural enabler. Lift `ThemeProvider` + `CssBaseline` out of `Catalog` into a new `AppShell`
component, convert `Catalog` from closing over the module-scoped `items` array to accepting
`items` as a prop, rename the DOM mount point from `catalog-root` to `app-root`, and replace
the `root.render(h(Catalog))` call with `root.render(h(AppShell))`. Per Gray Area A (Option 1),
remove the existing Jinja `<h2>App Catalog</h2>` and its subtitle `<p>` so the React region owns
the headings once Plan 15-03 inserts the "Browse by category" H2.

Purpose: Plan 15-03 introduces user-visible behavior — category cards, filter state, URL hash
sync, empty state, accessibility. Those additions assume (1) one `ThemeProvider` at the top of
the React tree so `Categories` and `Catalog` inherit the same MUI theme; (2) `Catalog` is
pure — it renders whatever `items` array it's handed, with no internal filter logic; (3) the
mount id reflects the fact that the root renders a whole shell, not just the catalog. This plan
performs those three prerequisites as a separate, independently-verifiable structural refactor.
The rendered page MUST remain pixel-identical to its post-15-01 state, modulo the removed
`App Catalog` heading and subtitle.

This plan MUST NOT:
- Introduce `useState`, `useEffect`, or add anything to the React destructure (that is Plan 15-03)
- Create `Categories` or `EmptyState` components (Plan 15-03)
- Parse `window.location.hash` or call `history.replaceState` (Plan 15-03)
- Add any filter logic — `AppShell` passes the full `items` array through to `Catalog`
- Add the "Browse by category" H2 or subtitle (Plan 15-03)
- Change card styling, card order, tag Chips, or the "Coming Soon" ribbon on any existing item
- Touch any file other than `index.html`

Pitfall 5 from PITFALLS.md (`ThemeProvider` must lift from `Catalog` to `AppShell`) is the
primary risk this plan mitigates. Grep-level verification of the pitfall is in the automated
check.
</objective>
