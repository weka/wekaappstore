**FastAPI `TemplateResponse` context attribute — pin `response.context`:**
`FastAPI.templating.Jinja2Templates.TemplateResponse(...)` returns a `starlette.templating._TemplateResponse` instance which exposes the rendering context dict as the public attribute `response.context`. This is the documented public Starlette API (see `starlette.templating._TemplateResponse.__init__` — `self.context = context`). There is no public `context_data` attribute on `_TemplateResponse`; tests SHOULD assert `response.context["credentials_by_type"]` directly. If at runtime `response.context` is absent, that is a real bug in Plan 01's `blueprint_detail` route handler (it constructed the response wrong), not a test-discovery problem — the test SHOULD fail loudly rather than papering over the missing attribute.

`get_auth_status` and `get_cluster_status` are synchronous functions in `main.py` that read in-cluster state. In tests they must be monkeypatched to return deterministic dicts: `monkeypatch.setattr(main, "get_auth_status", lambda: {"details": {"namespace": "test-ns"}})` etc.

`blueprint_detail` route signature: `async def blueprint_detail(request: Request, name: str)`. In tests, pass a `SimpleNamespace()` as the `request` argument — FastAPI's `TemplateResponse` accepts any object as `request` and stashes it on the response without invoking it.
