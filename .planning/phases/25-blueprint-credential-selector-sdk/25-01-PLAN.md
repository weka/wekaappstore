Output: A single refactored `main.py` with one new helper function, one route updated (`blueprint_detail`), and one route's inline block replaced with the helper call (`settings_page`).
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → FastAPI route (`GET /blueprint/{name}`) | Path parameter `name` enters via the route; the URL flows to template selection (`f"blueprint_{name}.html"`) and `app_map` lookup — pre-existing surface, unchanged in this plan. |
| FastAPI route → Kubernetes API server | Synchronous K8s SDK calls (`get_auth_status`, `CustomObjectsApi.list_namespaced_custom_object`) are made server-side via `asyncio.to_thread`. The plan adds one more such call site (`blueprint_detail`) — identical trust shape to `settings_page`. |
| FastAPI route → Jinja2 template (`templates/blueprint_*.html`) | Server-rendered context dict is passed to `templates.TemplateResponse`. The new key `credentials_by_type` is a `dict[str, list[dict]]` whose inner dicts come from `_build_credential_response_item` — a function that already filters secret material out of CR data (audited in Phase 23, test `test_build_credential_response_item_omits_secret_fields`). |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-25-01 | I (Information Disclosure) | `_get_credentials_by_type` return value injected into template context | mitigate | The helper builds items via `_build_credential_response_item` (`main.py:744`), which by Phase 23 design never reads `spec.secretRef.name` content or any raw Secret value — only metadata (`name`, `displayName`, `type`, `ready`, `endpoint` from `status.wekaEndpoint`, `lastSyncTime`, `derivedSecrets` names). No raw key material reaches the template context. Verified by existing test `test_build_credential_response_item_omits_secret_fields` (test_credentials_api.py:97-105) which asserts forbidden keys `{"key", "apiKey", "token", "secretRef", "NGC_API_KEY", "WEKA_API_TOKEN", "WEKA_API_USERNAME", "password"}` are absent from the item dict. |
| T-25-02 | D (Denial of Service) | Blueprint page load when K8s API is unreachable | mitigate | The helper catches `(ApiException, ConnectionError, TimeoutError)` and returns an empty dict-of-lists fallback. `blueprint_detail` never raises a 500 due to credentials lookup — page renders with empty `credentials_by_type` and Plan 02's macros degrade to hint mode. Behavior is locked by SDK-05 and the existing inline pattern (same exception triple) that `settings_page` has shipped with since Phase 24. |
| T-25-03 | T (Tampering) | Path parameter `name` in `/blueprint/{name}` route | accept | Pre-existing surface. The path is consumed only for `app_map` lookup, template-file existence check (`os.path.exists(os.path.join(TEMPLATES_DIR, preferred))`), and template selection. No filesystem traversal possible because `f"blueprint_{name}.html"` is concatenated as a basename and joined to `TEMPLATES_DIR` (no `..` handling in `name` could escape because `os.path.join` with a basename does not interpret `..`-prefixed values as parent — and if a malicious `name` like `../../etc/passwd` were submitted, `os.path.exists` would return False and the fallback `blueprint.html` would be selected). This plan does not change the route signature or the template-selection logic, so the existing risk posture is unchanged. |
| T-25-04 | I (Information Disclosure) | Operator/GUI logs | mitigate | API-08 (already shipped in Phase 22+23) forbids logging any raw key value. This plan adds no new logging statements; it only adds two `asyncio.to_thread` calls and one helper-function definition. No `print`, no `logging.info`, no exception-message echo of secret material is introduced. |
</threat_model>
