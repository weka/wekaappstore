`credentials_by_type` is a `dict[str, list[dict]]` with exactly these keys:
- `"nvidia-ngc"`: list of credential dicts (may be empty)
- `"huggingface"`: list of credential dicts (may be empty)
- `"weka-storage"`: list of credential dicts (may be empty)

Plan 01 filters each list to ready-only credentials at the helper level (single source of truth for SDK-02's "ready" contract). The macros therefore do NOT need any `selectattr('ready')` or `{% if cred.ready %}` guard — every dict in each list is already ready=True by the time the macro consumes it. The empty branch fires when the list is empty (either no CRs exist OR none are ready OR K8s is unreachable).

Each credential dict has these fields (only the bolded ones are used inside the macros):
- **`name`**: string — used as `<option value="...">`
- **`displayName`**: string — used as `<option>` visible text
- `type`: `"nvidia-ngc" | "huggingface" | "weka-storage"`
- `ready`: bool (always True for entries that reach the macro)
- **`endpoint`**: string or `None` — only set for `weka-storage`, sourced from `status.wekaEndpoint` (never from a raw Secret)
- `lastSyncTime`, `derivedSecrets`, `dockerSecretReady` (nvidia-ngc only), `error` (when not ready) — not consumed by the macros

- Form control Tailwind class string: `settings.html:688` — `w-full px-3 py-2 rounded-md bg-gray-800/70 border border-white/10 focus:outline-none focus:ring-2 focus:ring-[var(--weka-purple)] text-sm`

- Empty-state hint pattern: `settings.html:136-140` — `<p class="muted text-sm">` + `<a class="text-[var(--weka-purple)] underline hover:text-white">`

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| `WarpCredential` CR admin → server-side template context | An administrator with permission to create `WarpCredential` CRs supplies `spec.displayName`, `metadata.name`, and `spec.endpoint` strings. These flow through `_build_credential_response_item` (`main.py:744`) into `credentials_by_type` and then into the macro outputs (`<option>` text, `<option value>`, `data-endpoint`, `<input value>`). |
| Server-rendered HTML → browser DOM | Rendered HTML from the macros is delivered to the user's browser as part of the blueprint install page. If a credential field were rendered without escaping, attacker-controlled `displayName` containing `"><script>...</script>` or `" onerror="..."` would break out of the `<option>` element or `data-endpoint` attribute and execute in the user's session. |
| Client-side `warpSyncEndpoint` JS → endpoint `<input>` value | The inline script reads `selectEl.options[selectEl.selectedIndex].dataset.endpoint` and writes it to `epInput.value`. `dataset.endpoint` is the parsed DOM-attribute value — already HTML-decoded by the browser. Setting `epInput.value` is a property assignment (not innerHTML), so it cannot execute script. |

| T-25-05 | T (Tampering) / Cross-Site Scripting (Reflected/Stored) | `<option>` text and `<option value>` rendered from `cred.displayName` and `cred.name` in `credential_select` and `weka_storage_select` | mitigate | Jinja2 autoescape is enabled by default for `.html` files via `Jinja2Templates(directory=TEMPLATES_DIR)` at `main.py:184` (FastAPI default Jinja2 Environment uses `autoescape=select_autoescape(['html', 'xml'])`). | T-25-06 | T (Tampering) / Open Redirect via `data-endpoint` attribute | `data-endpoint="{{ cred.endpoint or '' }}"` on `<option>` elements in `weka_storage_select` and the initial `value="{{ credentials_by_type['weka-storage'][0].endpoint or '' }}"` on the endpoint `<input>` | mitigate | The `data-endpoint` value is sourced from `WarpCredential.status.wekaEndpoint`, which the operator (Phase 22) copies verbatim from `spec.endpoint` after the operator-side reconcile — `spec.endpoint` is admission-validated as a string by the CRD schema (Phase 21, CRD-04). Jinja2 attribute escaping neutralizes `"` and `>` characters that could break out of the attribute. The value is consumed only by (a) the browser writing it into `<input type="url">.value` as a string property assignment (no script execution possible), and (b) the user manually editing the URL input before submit.
