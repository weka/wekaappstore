# Phase 24 — UI Design Contract

> Visual and interaction contract for the Settings GUI overhaul. Generated by gsd-ui-researcher.
>
> **Stack constraint (locked):** server-rendered Jinja2 from FastAPI, vanilla HTML + Tailwind CSS via CDN, vanilla JS (no framework, no build step), Inter font via Google Fonts CDN, no component library, no icon library. This contract extends the existing visual language of `app-store-gui/webapp/templates/settings.html` — every new element MUST be implementable with utility classes already used by that file.

## Spacing Scale

The page uses Tailwind's default 4-px scale. All new elements MUST snap to multiples of 4. Existing chrome (`max-w-6xl mx-auto px-4 py-8 space-y-8`, `.card { p-5 }`, header `py-6`) is unchanged.

| Token | Value | Tailwind | Usage in this phase |
|-------|-------|----------|----------------------|
| xs | 4px | `gap-1`, `p-1` | Traffic-light dot to label gap, badge inner padding |
| sm | 8px | `gap-2`, `p-2`, `py-2` | Credential row internal padding, inline form field gaps, table cell `py-2` |
| md | 16px | `gap-4`, `p-4`, `mb-4` | Sub-section internal spacing, form field row spacing, overview panel grid gap |
| lg | 24px | `p-5`, `gap-6`, `mb-6` | Card padding (matches existing `.card { p-5 }`), gap between credential sub-sections |
| xl | 32px | `space-y-8`, `py-8` | Vertical rhythm between top-level `<section>` cards (already used by `main`) |
| 2xl | 48px | `mt-12` | Reserved — not used in this phase |
| 3xl | 64px | n/a | Reserved — not used in this phase |

Exceptions:
- Traffic-light dot is **10px** (`w-2.5 h-2.5`) to match the existing `.status-dot` rule at `settings.html:28`. This is the only non-multiple-of-4 dimension in the phase and is a direct continuation of the established status-indicator pattern.
- Utilisation progress bar height is **8px** (`h-2`). Filesystem-table mini bars: 6px (`h-1.5`) — exception justified by table row density; both are multiples of 2 and visually small enough that the 4-px rule is impractical.

## Component Inventory

The following composite components are introduced. Each is a vanilla HTML pattern composed of Tailwind utility classes — no JS framework, no third-party library, no shadcn registry.

### 1. `CredentialManagementSection` (top-level container)

- Outer: `<section class="card rounded-lg p-5">` (matches existing card pattern)
- Heading row: `<h3 class="font-semibold">Credential Management</h3>` + `<p class="muted text-sm">` sub-copy
- Three nested sub-section cards rendered in fixed order: NGC, HuggingFace, WEKA Storage (matches GUI-02 order)
- Position: **first** `<section>` inside `<main>`, directly after the `<h2>Settings</h2>` heading. All other sections shift down.

### 2. `CredentialTypeSubsection` (one per type)

- Outer: nested `<div class="card rounded-lg p-5">` (card-in-card)
- Header row: type title (`h4 font-semibold`) on the left + `[+ Add]` button on the right
- Credential list: `<ul role="list">` with one `<li>` per credential row
- Empty state: `<p class="muted text-sm">(none stored)</p>` rendered in place of the list when no credentials of this type exist
- Inline add-form slot: hidden by default, expands when `[+ Add]` is clicked, collapses on Save/Cancel

### 3. `CredentialRow` (one per credential, three visual states)

Layout: flex row, `py-2`, border-bottom `border-white/5` between rows. Left side = state dot + display name + state badge. Right side = `[Delete]` button (red-outline secondary).

Green state DOM:
```html
<li class="flex items-center justify-between py-2 border-b border-white/5 last:border-b-0">
  <div class="flex items-center gap-2">
    <span class="status-dot bg-green-500" aria-hidden="true"></span>
    <span class="text-sm">{display_name}</span>
    <span class="text-xs px-2 py-0.5 rounded-full border border-green-500/40 bg-green-500/10 text-green-400">Ready</span>
  </div>
  <button class="px-3 py-1 rounded-md text-sm border border-red-400/40 text-red-300 hover:bg-red-500/10" data-cred="{name}">Delete</button>
</li>
```
Amber: dot uses `bg-amber-500 animate-pulse`, badge says `Verifying…`, **NO Delete button** during polling (prevents race with operator reconcile).
Red: dot `bg-red-500`, no badge wrapper — instead a `text-red-400 text-xs` error reason rendered after the name. Delete button is present.

Accessibility: each row has `aria-live="polite"` on the badge span so screen readers announce state transitions. The dot is `aria-hidden="true"`; state semantics come from the badge text and an `aria-label` on the row.

### 4. `InlineAddForm` (one per credential type, max one open at a time)

NGC + HuggingFace form (2 fields):
- `Name` — `<input type="text" required>` (display name)
- `Key` — `<input type="password" required>`
- Buttons: `[Save]` (purple primary, disabled until both fields non-empty) + `[Cancel]` (outline secondary)

WEKA Storage form (4 fields):
- `Name` — `<input type="text" required>`
- `Username` — `<input type="text" required>`
- `API Token` — `<input type="password" required>`
- `Endpoint` — `<input type="url" required>` with client-side validation; Save remains disabled until `endpoint.validity.valid === true` AND all other fields are non-empty
- Buttons: `[Save]` + `[Cancel]`

Form layout: `<form class="mt-4 p-4 rounded-md bg-white/5 border border-white/10 space-y-3">` with grid for fields. Inputs use `w-full px-3 py-2 rounded-md bg-gray-800/70 border border-white/10 focus:outline-none focus:ring-2 focus:ring-[var(--weka-purple)] text-sm`.

Single-open-form invariant (GUI-03): clicking `[+ Add]` on type B while a form on type A is open MUST collapse the type-A form (discarding its unsaved input — no warning, since no value has been submitted to the backend yet; this is the simpler and safer choice for a credential form).

### 5. `WekaStorageOverviewPanel`

Outer `<section class="card rounded-lg p-5">`, rendered conditionally:

| Condition | Rendered content |
|-----------|------------------|
| Zero `weka-storage` credentials | Hint: `<p class="muted text-sm">No WEKA Storage credential configured. <a href="#weka-credentials" class="text-[var(--weka-purple)] underline hover:text-white">Add one above.</a></p>` (GUI-10, GUI-15) |
| ≥ 1 credential, loading | Centered spinner: `<div role="status" class="inline-block w-6 h-6 border-2 border-white/20 border-t-[var(--weka-purple)] rounded-full animate-spin"></div>` + label `<span class="muted text-sm">Loading WEKA cluster…</span>` |
| ≥ 1 credential, fetch error | Banner: `<div class="rounded-md border border-red-400/40 bg-red-500/10 p-4 text-sm text-red-300">{error message}</div>` — NO stale data shown, NO refresh button auto-retry (GUI-15) |
| ≥ 1 credential, success | Full panel (header + capacity row + filesystem table + backend grid) |

Header (success state):
- Title `<h3 class="font-semibold">WEKA Storage Overview</h3>` on the left
- Right side: credential `<select>` (only rendered when `weka_credentials.length > 1`; static `<span class="text-sm muted">{display_name}</span>` otherwise) + `[↺ Refresh]` button (outline-purple) + `"Last updated {relative_time}"` in `text-xs muted`
- "Last updated" displays the age of the underlying `fetchedAt` ISO timestamp from the API response, NOT the time the page was loaded (GUI-11). Formatter: `"<60s ago"`, `"2m ago"`, `"15m ago"`, `"1h ago"`, etc.

Capacity row (success state):
```html
<div class="grid grid-cols-1 md:grid-cols-3 gap-4 mb-4">
  <div class="card rounded-lg p-5">
    <div class="muted text-xs">Total</div>
    <div class="text-2xl font-semibold">{total_human}</div>
  </div>
  <div class="card rounded-lg p-5">
    <div class="muted text-xs">Used</div>
    <div class="text-2xl font-semibold">{used_human}</div>
  </div>
  <div class="card rounded-lg p-5">
    <div class="muted text-xs">Available</div>
    <div class="text-2xl font-semibold">{available_human}</div>
  </div>
</div>
<div class="space-y-1">
  <div class="flex justify-between text-xs muted">
    <span>Utilisation</span><span>{percent}%</span>
  </div>
  <div class="w-full h-2 rounded-full bg-white/5 overflow-hidden">
    <div class="h-full {bg-amber-500 if pct>=90 else bg-[var(--weka-purple)]}" style="width: {percent}%"></div>
  </div>
</div>
```
Human-readable formatter: bytes → `1.23 TiB`, `456 GiB`, `12.0 MiB` — TiB / GiB / MiB chosen via highest unit where value ≥ 1.0; one decimal of precision; "B" / "KiB" suppressed (any capacity that small is malformed and rendered as raw bytes for debugging).

Filesystem table (success state):
- `<table class="min-w-full">` matching the existing Blueprint table at line 206
- Columns: `Name`, `Total`, `Used`, `Utilisation` (header `text-xs uppercase text-white/70`)
- Sort: descending by `usedPercent` server-side OR client-side at first render
- Row utilisation cell: numeric percent + thin progress bar (`h-1.5`); bar fill colour switches to `bg-amber-500` when `usedPercent ≥ 90` (GUI-13)
- Max 20 rows initially. If `filesystems.length > 20`, a `<button class="mt-3 text-sm muted hover:text-white underline">Show all ({n}) ▾</button>` toggle appears at the bottom; clicking expands all rows; button label flips to `Show top 20 ▴`
- **No UUIDs displayed** (GUI-13) — only the human `name` field from the API response

Backend node grid (success state):
```html
<div class="mt-4">
  <h4 class="font-semibold text-sm mb-2">Backend Nodes ({count})</h4>
  <div class="grid grid-cols-2 sm:grid-cols-3 md:grid-cols-4 lg:grid-cols-6 gap-2">
    <div class="font-mono text-xs px-3 py-2 rounded-md bg-white/5 border border-white/10">{ip}</div>
    <!-- repeated -->
  </div>
</div>
```
IP filtering happens in the **backend** (already locked in Phase 23) — the GUI renders whatever `backendNodes[]` returns and does not filter loopback/link-local itself. If `backendNodes.length === 0`, render `<p class="muted text-sm">No backend node IPs reported.</p>`.

### 6. Polling state machine (GUI-07, locked)

State transitions for a newly-created credential row:

```
[Save clicked] → POST /api/credentials → 202 response
       │
       ▼
   AMBER state + setInterval(2000)
       │
       │   each tick: GET /api/credentials, find row by name
       │
       ├── ready: true            → GREEN state, clearInterval
       ├── error string present   → RED state, clearInterval
       ├── 30 seconds elapsed     → RED state with copy
       │                            "Verification timed out. The operator may
       │                             still be reconciling — refresh the page
       │                             to recheck."
       │                            clearInterval
       └── network failure        → keep amber, retry next tick; bubble final
                                    timeout into the same 30 s budget
```

Implementation: vanilla JS `setInterval` per amber row, tracked in a `Map<name, intervalId>` to allow cancellation. When the row reaches green/red, `clearInterval` is called and the interval is removed from the map.

A `Page Visibility API` check pauses polling when `document.hidden === true` and resumes on `visibilitychange`. This protects against runaway timers on backgrounded tabs.

## Accessibility Contract

- Every interactive element has a visible focus ring: `focus:outline-none focus:ring-2 focus:ring-[var(--weka-purple)]` (already used elsewhere on the page).
- Credential rows use `role="listitem"` (implicit via `<li>`); list container is `<ul role="list">`.
- Traffic-light dots have `aria-hidden="true"`; state is conveyed by the badge text content (no colour-only signalling).
- Polling rows have `aria-live="polite"` on the badge so transitions are announced.
- `Save` and `Delete` buttons have an `aria-label` that includes the credential display name when the visible label is generic (e.g. `aria-label="Delete credential ngc-prod"`).
- The browser-native `confirm()` modal for Delete satisfies destructive-action confirmation; users with screen readers receive the standard dialog announcement.
- Form inputs have associated `<label for="…">` tags; no placeholder-as-label pattern.
- Keyboard: `Tab` order follows DOM order (sub-section by sub-section, top-to-bottom); `Esc` inside an open add-form closes the form without saving (bind to the form container, not document-wide).

## Performance Contract

- Polling interval: 2000 ms (locked by GUI-07). Per-row only — no global poll loop. Max concurrent amber rows in realistic use: < 5.
- WEKA Overview Refresh button calls `GET /api/weka/overview?credential=<name>&bust=1` (the Phase 23 cache-bust query param). Default page-load fetch hits the 60 s cache.
- Page load injects `credentials_by_type` server-side via Jinja2 — initial render is one round-trip, no client-side fetch for the credential list on first paint.
- WEKA Overview data is fetched **after** initial render via a single `GET /api/weka/overview?credential=<default>` (no `bust=1`) when the panel is in success-target state. The "Loading" spinner is shown during this fetch.
- All polling intervals MUST be cleared on `beforeunload` to prevent leaks if the user navigates away.
