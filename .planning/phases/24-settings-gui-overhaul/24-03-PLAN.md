- `capacity`: `{totalBytes: number, usedBytes: number, availableBytes: number, usedPercent: number}`.
- `filesystems`: `Array<{name: string, totalBytes: number, usedBytes: number, usedPercent: number}>` (no `uid` field per GUI-13).

- `fetchedAt`: ISO-8601 timestamp string (e.g., `2026-06-12T15:04:05Z`).
- Failure: HTTP non-200 with `{ok: false, error: string}`. WEKA login failure → 502; SSRF guard rejection → 400; K8s API errors → 4xx/500.

     if (!currentWekaCredentialName) return;
     setWekaState('loading');
     const params = new URLSearchParams({
       credential: currentWekaCredentialName,
       namespace: DETECTED_NAMESPACE,
     });
     if (bust) params.set('bust', '1');
     try {
       const data = await get(`/api/weka/overview?${params.toString()}`);
       if (data && data.ok && data.data) {
         renderWekaSuccess(data.data);
         setWekaState('success');
       } else {
         renderWekaError(data && data.error ? data.error : 'Unknown error');
         setWekaState('error');
       }
     } catch (e) {
       renderWekaError(String(e));
       setWekaState('error');
     }
   }

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser ↔ `/api/weka/overview` | The endpoint proxies the WEKA REST API. An attacker who controls a WEKA cluster's filesystem name field could inject HTML/JS into the response. The backend whitelists fields (`_assemble_weka_overview` at main.py:1057-1064) but does NOT HTML-escape — escape happens at the DOM-write boundary on the client. |
| Browser ↔ `weka_storage_credentials` Jinja2 emit | The `displayName` and `endpoint` of every weka credential are emitted via `| tojson` into a JS literal. The `tojson` filter is XSS-safe in JS-string context. |
