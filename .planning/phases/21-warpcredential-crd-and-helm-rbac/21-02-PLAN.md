<objective>
Create `weka-app-store-operator-chart/templates/rbac.yaml` with a namespace-scoped `Role` and `RoleBinding` that grant the operator's service account create/patch/get/delete/list/watch on Secrets within the App Store namespace. Bump `Chart.yaml` version from 0.1.63 to 0.1.64.

The namespace-scoped Role follows least-privilege — write access to Secrets stays bounded to the weka-app-store namespace.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Helm chart → Kubernetes RBAC | Chart deploys a Role; Kubernetes enforces it at the API server |
| operator ServiceAccount → Secrets | The bound Role defines the exact write surface for the operator on Secrets |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-21-04 | Elevation of Privilege | Role scope | mitigate | Using kind: Role (not ClusterRole) ensures Secret CRUD is bounded to .Release.Namespace; cannot read/write Secrets in other namespaces |
| T-21-05 | Information Disclosure | Role verbs | accept | The operator needs full Secret CRUD to create derived secrets idempotently; read-only would not suffice for the reconciler use case |
| T-21-06 | Tampering | ClusterRole wildcard overlap | accept | The existing ClusterRole already grants secrets read/write cluster-wide via the wildcard rule in values.yaml; the new Role adds no new effective permissions — it explicitly documents the intended access surface for Phase 22 |
</threat_model>
