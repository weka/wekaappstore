## Summary

Phase 13 authors the complete Kubernetes manifest set to wire the MCP server as a sidecar into the existing OpenClaw Sandbox CR. The project has a validated topology from Phase 12 (TOPOLOGY.md) and an existing Sandbox CR in `k8s/agent-sandbox/openclaw-sandbox.yaml` that serves as the base for all modifications.

The Sandbox CR uses `agents.x-k8s.io/v1alpha1` and exposes `spec.podTemplate.spec` for pod configuration — the same fields as a plain Deployment pod spec. Phase 13 adds four things to this spec: an init container (openclaw.json generator), two new regular containers (MCP sidecar + git-sync), additional volumes (blueprint emptyDir, SKILL.md ConfigMap), and a `serviceAccountName`. The existing operator's ServiceAccount is not reused; a new `weka-mcp-server-sa` with a purpose-built ClusterRole is required.

All six manifest files are standalone YAML (not Helm templates) applied with `kubectl apply -n wekaappstore`. The existing pattern in this project for standalone RBAC is demonstrated in `weka-app-store-operator-chart/templates/deploy-app-store-gui.yaml` — ServiceAccount, ClusterRole, ClusterRoleBinding, and application manifests co-located. For Phase 13, these live as separate files under `k8s/agent-sandbox/`.

**Primary recommendation:** Author six YAML files: `mcp-rbac.yaml` (SA + ClusterRole + ClusterRoleBinding), `mcp-skill-configmap.yaml` (SKILL.md content), and update `openclaw-sandbox.yaml` (add init container, two new containers, volumes, serviceAccountName). Use the git-sync v4.5.0 image already in the project's Helm chart with `--one-time=false` for continuous sync. Use the K8s native sidecar pattern (`restartPolicy: Always`) for init containers only if the agent-sandbox operator version supports KEP-753; otherwise use regular containers with readiness dependency.

### git-sync v4.5.0 Arg Syntax (HIGH confidence — from existing project file)

```bash
# One-shot pattern (GUI deployment — initContainer)
--repo=https://github.com/weka/warp-blueprints.git
--root=/manifests
--link=manifests
--one-time

# Continuous pattern (Phase 13 — regular container, no --one-time)
--repo=https://github.com/weka/warp-blueprints.git
--root=/blueprints
--link=blueprints
--period=60s    # sync interval (default is 10s; 60s is reasonable for catalog)
```

The `--link` arg sets the symlink name inside `--root`. The application reads via the symlink path: `<root>/<link>/` (e.g., `/blueprints/blueprints/`). The GUI uses `subPath: manifests` in the volumeMount to dereference this. For the MCP sidecar, `BLUEPRINTS_DIR` should point to the dereferenced path.

### Pattern 1: Sandbox CR Pod Spec Modification

**What:** The Sandbox CR `spec.podTemplate.spec` accepts the same fields as a Kubernetes Pod spec. To add sidecar containers, add entries to `spec.podTemplate.spec.containers[]`. To add init containers, add to `spec.podTemplate.spec.initContainers[]`. Volumes go in `spec.podTemplate.spec.volumes[]`.

**When to use:** This is the only pattern available — Phase 13 modifies the Sandbox CR, not a Deployment.

**Existing Sandbox CR structure (source: `k8s/agent-sandbox/openclaw-sandbox.yaml`):**

```yaml
apiVersion: agents.x-k8s.io/v1alpha1
kind: Sandbox
metadata:
  name: openclaw-sandbox
spec:
  podTemplate:
    spec:
      nodeSelector: ...
      tolerations: ...
      securityContext:
        fsGroup: 1000
      containers:
        - name: openclaw
          # existing openclaw container
      volumes:
        - name: openclaw-config
          emptyDir: {}
  volumeClaimTemplates:
    - metadata:
        name: openclaw-workspace
```

**Required additions for Phase 13:**

```yaml
spec:
  podTemplate:
    spec:
      serviceAccountName: weka-mcp-server-sa   # ADD
      initContainers:                            # ADD
        - name: openclaw-json-generator
          # writes openclaw.json to /home/node/.openclaw/
      containers:
        - name: openclaw        # EXISTING — add volumeMount for SKILL.md ConfigMap
        - name: weka-mcp-sidecar  # ADD
        - name: git-sync          # ADD
      volumes:
        - name: openclaw-config   # EXISTING (shared emptyDir for openclaw.json)
        - name: blueprints        # ADD (emptyDir for git-sync)
        - name: skill-md          # ADD (ConfigMap volume)
  volumeClaimTemplates: ...       # EXISTING (no change)
```

### Pattern 2: Init Container for openclaw.json Generation

**What:** A minimal busybox init container runs a shell script that writes `openclaw.json` to the shared emptyDir volume before any other container starts.

**When to use:** K8S-05 requirement; init containers run to completion before regular containers start.

**Implementation (source: TOPOLOGY.md + existing `mcp-server/openclaw.json` structure):**

```yaml
initContainers:
  - name: openclaw-json-generator
    image: busybox:1.36
    command:
      - sh
      - -c
      - |
        cat > /home/node/.openclaw/openclaw.json << 'EOF'
        {
          "name": "weka-app-store-mcp",
          "description": "MCP server for the WEKA App Store. Provides tools for inspecting cluster and WEKA storage resources, browsing blueprint catalogs, validating WekaAppStore YAML manifests, applying blueprints to the cluster, and monitoring deployment status.",
          "transport": "streamable-http",
          "url": "http://localhost:${MCP_PORT:-8080}/mcp",
          "skill": "${SKILL_MD_PATH:-/home/node/.openclaw/SKILL.md}"
        }
        EOF
    env:
      - name: MCP_PORT
        value: "8080"
      - name: SKILL_MD_PATH
        value: "/home/node/.openclaw/SKILL.md"
    volumeMounts:
      - name: openclaw-config
        mountPath: /home/node/.openclaw
```

**IMPORTANT:** Shell variable expansion inside a heredoc requires careful handling. The safer approach is to use `printf` or `echo` with explicit values rather than relying on variable expansion within single-quoted heredoc. Use env vars for the values that change (MCP_PORT), and write them directly via printf:

```yaml
command:
  - sh
  - -c
  - |
    printf '{"name":"weka-app-store-mcp","transport":"streamable-http","url":"http://localhost:%s/mcp","skill":"%s"}\n' \
      "${MCP_PORT:-8080}" "${SKILL_MD_PATH:-/home/node/.openclaw/SKILL.md}" \
      > /home/node/.openclaw/openclaw.json
```

For the full JSON structure (including the tools list), the init container must write the full `openclaw.json`. The planner must decide whether to embed the full tool descriptions or use the minimal structure (name, transport, url, skill) that OpenClaw needs for registration. Given the locked decision says "openclaw.json includes both MCP server config (transport, url) AND SKILL.md path reference," the minimal structure is correct for the init container — the tools list is optional for registration.

**Note on tools field:** The existing `mcp-server/openclaw.json` includes a detailed `tools` array. This is likely for documentation/reference, not required by OpenClaw at registration time. Research should confirm, but the locked decision says the init container writes config "from env vars" — a full tools list cannot be generated from env vars alone, so the init container should write the minimal registration config only.

### Pattern 3: MCP Sidecar Container Spec

**What:** Regular Kubernetes container with readiness probe on `/health` at port 8080.

**Source:** MCP server exposes `EXPOSE 8080` and `/health` endpoint (XPORT-03 complete, Phase 11).

```yaml
- name: weka-mcp-sidecar
  image: wekachrisjen/weka-app-store-mcp  # CI-published; no tag pinned in dev
  imagePullPolicy: Always
  ports:
    - containerPort: 8080
      name: mcp-http
      protocol: TCP
  env:
    - name: MCP_TRANSPORT
      value: "http"
    - name: MCP_PORT
      value: "8080"
    - name: BLUEPRINTS_DIR
      value: "/blueprints/blueprints"   # git-sync writes to /blueprints/<link>
    - name: KUBERNETES_AUTH_MODE
      value: "incluster"
  readinessProbe:
    httpGet:
      path: /health
      port: 8080
    initialDelaySeconds: 5
    periodSeconds: 5
    failureThreshold: 3
  livenessProbe:
    httpGet:
      path: /health
      port: 8080
    initialDelaySeconds: 15
    periodSeconds: 15
  resources:
    requests:
      cpu: "100m"
      memory: "256Mi"
    limits:
      cpu: "500m"
      memory: "512Mi"
  securityContext:
    runAsNonRoot: true
    runAsUser: 10001          # matches Dockerfile useradd -u 10001 mcpuser
    allowPrivilegeEscalation: false
    capabilities:
      drop:
        - ALL
  volumeMounts:
    - name: blueprints
      mountPath: /blueprints
    - name: openclaw-config
      mountPath: /home/node/.openclaw
      readOnly: true
```

### Pattern 4: Startup Ordering

**What:** The locked decision requires MCP sidecar readiness before OpenClaw reads openclaw.json.

**Options researched:**

**Option A — Regular containers with dependency via readiness (RECOMMENDED for agent-sandbox compatibility):**
- Init container (openclaw.json generator) runs first — completes before any regular container starts (K8s guarantee)
- Regular containers start in parallel: openclaw, weka-mcp-sidecar, git-sync
- OpenClaw reads openclaw.json at startup and discovers the MCP server URL
- MCP sidecar readiness probe prevents OpenClaw from getting a 503 when it first connects
- No K8s native sidecar feature required — works with any Kubernetes version

**Option B — K8s Native Sidecar (KEP-753, available since K8s 1.29 stable):**
- Set `restartPolicy: Always` on an init container to make it a "sidecar init container"
- Sidecar init containers start before regular containers AND stay running
- OpenClaw (regular container) only starts after all sidecar init containers are ready
- Requires EKS 1.29+ and the agent-sandbox operator to not strip `restartPolicy` from init container specs

**Recommendation for Phase 13:** Use Option A (regular containers). The agent-sandbox operator version (v0.2.1) is experimental; its handling of `restartPolicy: Always` on init containers is unknown. Option A provides the same ordering guarantee via standard K8s init container behavior + readiness probe, without depending on operator behavior.

**Startup sequence with Option A:**
1. Init container `openclaw-json-generator` runs → writes openclaw.json → exits 0
2. Regular containers start: `openclaw`, `weka-mcp-sidecar`, `git-sync` (parallel)
3. `weka-mcp-sidecar` readiness probe gates its `Ready` status
4. OpenClaw reads openclaw.json and attempts to connect to `http://localhost:8080/mcp`
5. If OpenClaw respects the MCP server's readiness before tool registration, ordering is correct
6. git-sync pulls blueprints independently; BLUEPRINTS_DIR is available once first sync completes

**Known risk:** If OpenClaw immediately retries tool registration at startup before the MCP sidecar is ready, it may log connection errors but should recover once the sidecar becomes ready. The `--allow-unconfigured` flag on OpenClaw suggests it tolerates missing MCP servers at startup.

### Anti-Patterns to Avoid

- **Using `serviceAccountName` in the wrong location:** In the Sandbox CR, `serviceAccountName` goes under `spec.podTemplate.spec.serviceAccountName`, not at the Sandbox CR metadata level.
- **Hardcoding namespace in ClusterRoleBinding:** The `subjects[].namespace` field must match the namespace where the ServiceAccount is created (`wekaappstore`). This is a named field requirement even for ClusterRoleBindings.
- **git-sync as initContainer with `--one-time`:** The locked decision requires continuous sync. Use git-sync as a regular container without `--one-time`.
- **Mounting openclaw-config emptyDir as ReadWrite in MCP sidecar:** The openclaw.json generator writes to this volume in the init container; MCP sidecar only needs to read from it. Set `readOnly: true` in the sidecar's volumeMount.
- **BLUEPRINTS_DIR pointing to git-sync root (not symlink):** git-sync creates a symlink `<link>` inside `<root>`. The actual content is at `<root>/<link>/`. If `--root=/blueprints` and `--link=blueprints`, then `BLUEPRINTS_DIR=/blueprints/blueprints`. The existing GUI deployment uses `subPath: manifests` to avoid the double-directory issue. Choose one approach consistently.
- **Omitting `fsGroup: 1000` for git-sync volume:** The pod-level `securityContext.fsGroup: 1000` is already set in the existing Sandbox CR. This ensures the emptyDir volumes created for blueprints are group-writable. git-sync runs as a different user (typically uid 65533 in v4.5.0) — this may require either setting `runAsUser` on git-sync or using `fsGroup` carefully. Check git-sync v4.5.0 default user.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Continuous blueprint sync | Shell cron or custom sync container | `registry.k8s.io/git-sync/git-sync:v4.5.0` | Already in this project; handles exponential backoff, symlink swap, sparse checkout, auth |
| YAML config generation at pod start | Python script in MCP image (startup cost) | `busybox:1.36` init container with `printf`/`cat` | Faster pull, no Python overhead, purpose-built for one-shot file writes |
| SKILL.md versioning | Git submodule or bespoke sync | K8s ConfigMap | Static file; ConfigMap is the standard K8s pattern for < 1MB config files |
| Startup ordering enforcement | Sleep loops, custom readiness scripts | K8s init containers + readiness probes | K8s guarantees init containers complete before regular containers start |

**Key insight:** git-sync v4.5.0 handles all the edge cases of git repo syncing (partial clone, auth, retry) that a hand-rolled solution would need to reimplement.

## Common Pitfalls

### Pitfall 1: git-sync symlink path confusion

**What goes wrong:** `BLUEPRINTS_DIR` is set to `/blueprints` (the root) instead of `/blueprints/blueprints` (the symlink), so the MCP server sees an empty or wrong directory.

**Why it happens:** git-sync creates `<root>/<link>` as a symlink. The content is not directly at `<root>/`. The existing GUI deployment works around this with `subPath: manifests` in the volumeMount, which dereferences the symlink at mount time.

**How to avoid:** Either use `subPath: <link>` in the MCP sidecar's volumeMount for the blueprints volume, OR set `BLUEPRINTS_DIR` to `<root>/<link>`. Recommend using `subPath` approach (matches existing project pattern):

```yaml
# git-sync container
- name: git-sync
  volumeMounts:
    - name: blueprints
      mountPath: /blueprints
# MCP sidecar container
- name: weka-mcp-sidecar
  env:
    - name: BLUEPRINTS_DIR
      value: "/app/blueprints"
  volumeMounts:
    - name: blueprints
      mountPath: /app/blueprints
      subPath: blueprints    # dereferences git-sync symlink
```

**Warning signs:** `list_blueprints` returns `count: 0` or MCP server logs show "BLUEPRINTS_DIR not found" at startup.

### Pitfall 2: ClusterRoleBinding subject namespace mismatch

**What goes wrong:** `kubectl apply` succeeds but the MCP sidecar gets 403 Forbidden on all K8s API calls.

**Why it happens:** ClusterRoleBindings require `subjects[].namespace` to match the namespace where the ServiceAccount actually lives. A mismatch means the binding is syntactically valid but functionally broken — Kubernetes doesn't validate this at apply time.

**How to avoid:** Always set `subjects[0].namespace: wekaappstore` in the ClusterRoleBinding. This matches the `kubectl apply -n wekaappstore` deployment target.

**Warning signs:** `kubectl logs <mcp-sidecar>` shows `[403] Forbidden` on inspect_cluster calls. Check with: `kubectl auth can-i list nodes --as=system:serviceaccount:wekaappstore:weka-mcp-server-sa`.

### Pitfall 3: Init container writes to wrong path or wrong volume

**What goes wrong:** OpenClaw starts but has no openclaw.json; MCP server is never discovered.

**Why it happens:** The init container writes to a path that is not the shared emptyDir, or writes to the correct volume but at the wrong path inside it. OpenClaw reads from `/home/node/.openclaw/openclaw.json` (confirmed in TOPOLOGY.md).

**How to avoid:** Init container volumeMount must use `name: openclaw-config` (the existing emptyDir) at `mountPath: /home/node/.openclaw`. The write target is `/home/node/.openclaw/openclaw.json`.

**Warning signs:** `kubectl logs openclaw-sandbox -c openclaw-json-generator` shows successful write but `kubectl exec openclaw-sandbox -c openclaw -- cat /home/node/.openclaw/openclaw.json` returns "No such file".

### Pitfall 4: MCP sidecar user (uid 10001) cannot read blueprints volume

**What goes wrong:** MCP server starts but `list_blueprints` returns empty even after git-sync has synced content.

**Why it happens:** git-sync v4.5.0 runs as uid 65533 by default and creates files owned by that uid. The MCP sidecar runs as uid 10001. The pod-level `fsGroup: 1000` controls supplemental group ownership, not the owner. If the blueprints emptyDir files are 0600 (only owner-readable), the MCP sidecar cannot read them.

**How to avoid:** Explicitly set `securityContext.runAsUser: 65533` on the git-sync container to match the image default, and rely on `fsGroup: 1000` to make the volume group-writable. Alternatively, add `securityContext.runAsGroup: 1000` to git-sync so files are group-readable. Test with `kubectl exec` after deployment.

**Warning signs:** `kubectl exec <pod> -c weka-mcp-sidecar -- ls -la /blueprints/blueprints/` shows files owned by a different uid with mode 600.

### Pitfall 5: openclaw.json shell heredoc variable expansion

**What goes wrong:** The `url` field in openclaw.json contains the literal string `${MCP_PORT:-8080}` instead of `8080`.

**Why it happens:** Single-quoted heredoc (`<< 'EOF'`) prevents shell variable expansion. Double-quoted or unquoted heredoc (`<< EOF`) allows expansion but creates issues with JSON quotes and special characters.

**How to avoid:** Use `printf` with explicit format string instead of heredoc, or use `envsubst` if available in the busybox image. The safest approach for busybox is `printf`:

```sh
printf '{"name":"weka-app-store-mcp","transport":"streamable-http","url":"http://localhost:%s/mcp","skill":"%s"}\n' \
  "${MCP_PORT}" "${SKILL_MD_PATH}" > /home/node/.openclaw/openclaw.json
```

**Warning signs:** `kubectl exec <pod> -c openclaw -- cat /home/node/.openclaw/openclaw.json` shows literal `${...}` in the url field.

- `--one-time` flag in git-sync for continuous sync use cases: Only appropriate for init containers; continuous delivery requires omitting this flag.

## Open Questions

1. **Does OpenClaw retry MCP tool discovery after startup?**
   - What we know: OpenClaw uses `--allow-unconfigured` flag, suggesting it tolerates missing tools
   - What's unclear: Whether it retries the `http://localhost:8080/mcp` connection after initial failure or only reads openclaw.json once at startup
   - Recommendation: Assume it reads once at startup. The readiness probe on the MCP sidecar is the safety net. If OpenClaw connects before the sidecar is ready, the `--allow-unconfigured` flag prevents a crash. Manual testing in Phase 14 will reveal actual behavior.

2. **Does the agent-sandbox operator v0.2.1 support K8s native sidecar containers (restartPolicy: Always on initContainers)?**
   - What we know: The operator is experimental (v0.2.1); native sidecars require K8s 1.29+
   - What's unclear: Whether the operator passes through `restartPolicy` on init containers or strips it
   - Recommendation: Use regular containers (Option A) for Phase 13. If the operator is confirmed to support KEP-753, Phase 14 can migrate to native sidecars.

3. **Does OpenClaw's openclaw.json `skill` field accept an absolute file path?**
   - What we know: The existing `mcp-server/openclaw.json` has `"skill": "mcp-server/SKILL.md"` (relative path from repo root)
   - What's unclear: Whether OpenClaw resolves this relative to its working directory or requires an absolute path in the container
   - Recommendation: Use an absolute path (`/home/node/.openclaw/SKILL.md`) to avoid working-directory ambiguity. Monitor `kubectl logs <openclaw>` for any "skill file not found" warnings.

4. **Does the NemoClaw ConfigMap schema require a specific `mcpServers` key structure?**
   - What we know: STATE.md gate says "NemoClaw ConfigMap schema for mcpServers.url must be verified against the deployed version"
   - What's unclear: The deployed OpenClaw version may use a different schema key than `mcpServers`
   - Recommendation: The TOPOLOGY.md already documents the correct structure as `{mcpServers: {weka-app-store: {transport: streamable-http, url: http://localhost:8080/mcp}}}`. This is what the init container should write. Validate post-deployment with `kubectl exec openclaw-sandbox -c openclaw -- cat /home/node/.openclaw/openclaw.json`.
