<task type="checkpoint:human-verify" gate="blocking">
  <name>Task 2: Conduct E2E chat session through SKILL.md steps 1-12</name>
  <what-built>
OpenClaw Web UI is accessible at http://localhost:18789. The MCP sidecar is healthy and all 8 tools are registered. Pre-chat evidence has been captured. The agent is ready to chat.
  </what-built>
  <how-to-verify>
Open http://localhost:18789 in your browser. This is the OpenClaw Web UI where you will chat with the agent.

**Conduct the full SKILL.md workflow by asking the agent to deploy the OSS Rag blueprint:**

Suggested opening prompt: "I'd like to deploy the OSS Rag blueprint. Please inspect the cluster and WEKA storage first to check if we have enough resources."

The agent should follow SKILL.md steps 1-12 automatically:
1. **inspect_cluster** — Agent reports GPU, CPU, RAM, namespaces, storage classes
2. **inspect_weka** — Agent reports WEKA capacity and filesystems
3. **list_blueprints** — Agent lists available blueprints; OSS Rag appears
4. **get_blueprint** (OSS Rag) — Agent describes the blueprint components
5. **get_crd_schema** — Agent reads the CRD schema (may be silent)
6. **Generate YAML** — Agent creates a WekaAppStore manifest
7. **validate_yaml** — Agent validates the manifest
8. **Retry loop** — Agent fixes any validation errors (may not be needed)
9. **Re-inspect cluster** — Agent re-checks resources before apply
10. **Present plan** — Agent shows you the YAML and asks for approval
11. **apply** — After you approve, agent applies the CR (say "proceed" or "yes, apply it")
12. **status** — Agent polls deployment status until Ready or Failed

**Evidence to capture during chat:**
- Copy the entire chat exchange (all messages) into a text file
- Note the timestamps if visible
- Pay attention to: Does the GPU/CPU data match what kubectl shows? Does OSS Rag appear in the blueprint list? Does the CR get created? Does the operator fully reconcile?

**After the chat session:**
- Save the chat transcript to `evidence/chat-transcript-e2e.md`
- Run: `bash scripts/capture-e2e-evidence.sh --post wekaappstore`

**Pass criteria for each requirement:**
- E2E-01 PASS: Agent returned real cluster data (GPU count, CPU cores, memory) that matches evidence/e2e-01-nodes.txt
- E2E-02 PASS: Agent listed blueprints and OSS Rag was in the catalog
- E2E-03 PASS: `kubectl get wekaappstores -n wekaappstore` shows the applied CR
- E2E-04 PASS: Agent reported deployment success AND `evidence/e2e-04-status.txt` shows appStackPhase=Ready (or all pods Running)

**If the operator takes a long time (>5 minutes):** This is expected for OSS Rag (GPU + WEKA storage). The agent should poll status automatically. Wait up to 15 minutes. Check `kubectl get pods -n wekaappstore -w` in a separate terminal to watch progress.
